# Inspection Patterns

## Default Order

1. 先看顶层目录和配置文件，例如 `pyproject.toml`、`package.json`、`requirements.txt`、`README`。
2. 再定位入口文件、主循环、服务启动点或路由注册点。
3. 然后用 `grep` 搜索关键标识符，把范围缩到最小。
4. 最后再读取关键文件中的必要片段。

## Budget Heuristic

1. 一个模块侦察任务里，目录树脚本或 `Get-ChildItem` 一般只需要 1 次。
2. 入口分析通常只需要 1 个入口文件 + 1 到 3 个关键支撑文件。
3. 如果已经能回答“模块职责是什么、入口在哪里、关键文件有哪些、依赖了谁”，就应收口，不必把整个目录读完。
4. 如果开始重复读同一个文件，说明应该改成总结结论或只补一条真正缺失的证据。

## When To Read More

- 用户要求解释某个函数、类或模块职责时，再扩大 `read` 范围。
- 用户准备修改代码时，再补充调用链、配置依赖和测试位置。
- 用户只要一个快速结论时，不要展开成全量代码漫游。

## Evidence Rules

- 目录结构结论可以来自 `glob`、目录树脚本或明确的路径搜索结果。
- 代码行为结论必须来自 `read` 或足够直接的搜索证据。
- 具体字段、函数列表、返回值、参数细节，必须来自直接读取对应定义位置；不能凭模块名或局部片段推断后写成事实。
- 若证据只足以支持“高层职责说明”，就输出高层职责说明，不要补写未经验证的低层细节。
- 不确定时明确说“当前证据只能说明什么，还不能说明什么”。
