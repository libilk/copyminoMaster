# Report Structure Guide

## Recommended Order

1. 先写一句总述，直接回答用户最关心的问题。
2. 再展开关键发现，只保留和任务目标有关的证据。
3. 如果存在风险、假设或未验证项，单独列出。
4. 如果用户需要下一步建议，再补行动建议。

## Good Patterns

- 问题明确时：总述 -> 关键证据 -> 结论
- 任务较复杂时：背景 -> 发现 -> 风险/缺口 -> 建议
- 涉及代码时：文件位置 -> 行为说明 -> 影响判断

## Avoid

- 只讲执行过程，不回答用户问题
- 证据不足却给出过度确定的结论
- 把“可能”“推测”“通常”写成“已经确认”
